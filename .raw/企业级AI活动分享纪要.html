ThinkingAI CTO 周津 — 产品架构与Agent落地实践

核心洞察

"让Agent学会协作，才是企业级Agent真正要解决的问题。"

过去几十年我们解决了"人的协作"（工作流/SOP、跨部门协作、组织流程），现在每个人有了一个Agent（客服Agent、运营Agent、投放Agent、分析Agent），但Agent之间存在孤岛·断点·无缝隙的问题。

从"装个Agent"到"用好Agent"的四重奏

  1. 业务层卡点 — Agent孤立无协作，每个Agent各行其是缺少协同
  2. 数据汇聚卡点 — 多模态全域数据汇聚难（行为·交互·多模态，Agent缺的是眼睛）
  3. 知识蒸馏卡点 — 数据到知识的蒸馏难（需要完整的采集·治理·资产体系）
  4. 安全卡点 — 安全治理缺位（权限·脱敏·合规·审计一个都不能少）

Agent的能力上限，由数据决定

"大模型给了Agent推理的能力，工程框架给了Agent执行能力，但Agent要'做对了'，靠的是对业务数据的深度理解。"
  • 没有数据的Agent = 通用聊天机器人
  • 有数据的Agent = 真正能驱动业务的专家

Agent智能监控与评测体系

三维诊断（贵·慢·错）：

  • 贵 = Token成本（识别无效Token消耗）
  • 慢 = 执行延迟（定位链路瓶颈）
  • 错 = 幻觉逻辑（过度思考 + 无效循环 + 上下文遗忘）

评测能力：

  • Agent执行沙箱（隔离环境安全试跑，不影响生产）
  • Agent A/B灰度能力（新旧Agent可灰度释放对比）
  • 数据准确性与口径一致性（同样的问题要有同样的答案）
  • 幻觉检测（防止Agent胡说八道）
关键认知：Agent上线不是终点，持续运营与治理，才决定它能跑多远。

三、Andrew的思考与我们的启发

Agent角色分工思考

"我们的Agent，有哪些是来提问的，哪些来干活的，哪些来验证的？"
类型作用例子
提问型Agent发现问题、提出假设、触发分析数据分析Agent（异常检测）、需求洞察Agent
干活型Agent执行任务、自动操作、交付结果智能运营Agent、广告投放Agent、内容生成Agent
验证型AgentA/B测试、效果评估、幻觉检测A/B实验Agent、监控评测Agent、口径一致性Agent

生产系统稳定性保障

"智能体运行时要保证生产系统中一个挂了不会影响全局。"
  • 需要设计Agent的熔断机制和降级策略
  • 参考AWS的架构: 隔离环境安全试跑 → 灰度释放 → 全量上线
  • 每个Agent应该有独立的运行时和可观测监控

A2A 协议选型思考：MCP vs Google A2A

  • MCP（Model Context Protocol）：Anthropic推出的开放标准，侧重Agent与外部工具/数据源的连接，已被AWS、ThinkingAI等广泛采纳，生态活跃度高
  • Google A2A（Agent-to-Agent）：Google主导的Agent间通信协议，侧重多Agent之间的协作编排，飞书、网易机会在探索适配场景

当前判断：

  1. 两者不是互斥关系，而是互补 — MCP解决"Agent如何调用工具"，A2A解决"Agent如何与其他Agent对话"
  2. 短期优先建议以MCP为主接入数据源和工具链，同时关注A2A进展为未来的多Agent编排做准备
  3. 企业自建Agent平台时，抽象一层协议通道层，避免被单一协议锁定

关键趋势判断

  1. 数据资产 > 模型能力：多家共识 — Agent的上限由数据决定，不是由模型决定
  2. 多Agent协作 > 单Agent能力：企业级Agent的核心问题是协作，不是单个Agent的IQ
  3. 知识库是基础设施：网易、ThinkingAI都强调知识库建立是重中之重
  4. 人机分工要明确：人定目标和边界，Agent在边界内自主运行
  5. 从短期ROI到长期战略价值：AI创新初期不应过度追求精确ROI计算
